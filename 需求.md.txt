› 总结该项目到目前为止所有的明确需求、逻辑以供之后做app或其他软件使用


• 以下是目前已经明确下来的项目需求和业务逻辑总结，可作为后续做 App、正式系统或其
  它端的产品说明基础。

  项目定位
  酒店客需工单看板 Demo，用于演示客人电话、前台录入或 AI 识别后的需求，如何形成工
  单，并让前台、员工、主管实时看到：

  - 有没有人接单
  - 谁在处理
  - 有没有完成
  - 是否超时
  - 哪些部门/员工处理效率更高

  第一版 AI 识别只使用本地规则，不接真实大模型，不接讯飞。未来可接讯飞、阿里云、
  腾讯云等语音转写服务。

  技术要求

  - Next.js App Router
  - TypeScript
  - Tailwind CSS
  - shadcn/ui 风格组件
  - lucide-react 图标
  - Supabase PostgreSQL
  - Supabase Realtime
  - Vercel 可部署
  - 无复杂登录，Demo 阶段员工通过下拉选择身份
  - 没有 Supabase 配置时，本地 Demo 模式不能白屏

  页面入口

  - / 首页，展示四个入口
  - /frontdesk 前台看板
  - /staff 员工手机端
  - /manager 主管看板
  - /ai-call AI 来电模拟页

  核心状态

  pending      // 待接单
  accepted     // 已接单
  processing   // 处理中
  completed    // 已完成
  timeout      // 已超时
  cancelled    // 已取消

  前台看板中，业务展示不应机械按数据库状态分组，而应按前台关心的问题分：

  - 未完成
  - 未接单
  - 已接单
  - 已完成
  - 已超时

  “已接单”在前台语义中包含：

  accepted + processing

  更稳妥的判断是：

  有 assignee_id 或 assignee_name => 已接单
  没有负责人 => 未接单

  部门

  housekeeping // 客房部
  engineering  // 工程部
  frontdesk    // 前台
  food         // 餐饮部
  manager      // 主管

  优先级

  low
  normal
  high
  urgent

  前台看板逻辑
  前台页面是最重要页面，适合 13-15 寸电脑屏展示。

  顶部 5 个数据卡片既是统计也是筛选按钮：

  未完成
  未接单
  已接单
  已完成
  已超时

  不再额外增加一排重复筛选按钮。

  顶部卡片视觉：

  - 全部白底
  - 选中卡片有明显边框
  - 未完成数字深色
  - 未接单数字橙色
  - 已接单数字蓝色
  - 已完成数字灰色
  - 已超时数字红色

  点击顶部卡片后筛选：

  未完成 -> 所有未完成工单
  未接单 -> 未完成且没有负责人
  已接单 -> 未完成且已有负责人
  已完成 -> completed 工单
  已超时 -> 未完成且已超时工单

  列表不再分区，统一列表展示。

  普通筛选支持排序：

  紧要优先
  最新优先
  最早优先

  紧要优先逻辑：

  先看是否超时，超时优先级最高
  再看等待时间，等待越久越靠前

  最新优先：

  created_at 倒序

  最早优先：

  created_at 正序

  点击“已超时”时：

  - 仍保留排序选择
  - 默认切换为“最早优先”
  - 因为超时工单应优先处理等待最久的

  工单列表布局：

  - 大屏尽量一行显示更多信息
  - 当前目标：大屏最多三列
  - 工单卡片不再强制分为“未接单超时 / 已接单超时 / 未接单未超时 / 已接单未超时”四
    个区

  工单卡片展示字段
  工单卡片直接展示：

  - 房号
  - 需求摘要
  - 部门中文名
  - 来源：人工录入 / AI 来电
  - 状态
  - 优先级
  - 负责人
  - 等待时间
  - 完成耗时
  - 是否超时，通过颜色和状态体现

  需求摘要展示规则：

  - 默认固定两行高度
  - 一行需求也占两行空间，避免卡片信息错位
  - 超过两行截断
  - 鼠标悬停显示完整需求文本

  卡片颜色规则：

  超时：红色边框 + 浅红背景
  未接单未超时：橙色边框 + 浅橙背景
  已接单/处理中未超时：蓝色边框 + 浅蓝背景
  已完成：灰色边框 + 浅灰背景，整体弱化
  已取消：深灰/弱化

  不再额外显示右上角“超时”角标，因为状态标签和红色卡片已经足够。

  超时逻辑
  不应只依赖数据库 status = timeout。

  前端应动态判断：

  当前时间 > due_at 且 status 不是 completed/cancelled => 超时

  因此即使数据库状态还是 pending 或 processing，只要超过 SLA，也要在前台显示为超
  时。

  员工端逻辑
  员工端适合手机浏览器。

  员工通过下拉选择身份，不做登录。

  员工端应该显示：

  - 本部门待接单工单
  - 自己已接单/处理中工单
  - 自己今日已完成工单

  操作：

  pending 工单 -> 接单
  processing/accepted 工单 -> 完成

  员工接单后：

  - 写入负责人
  - 写入接单时间
  - 前台实时显示负责人和处理中/已接单状态

  员工完成后：

  - 写入完成时间
  - 前台实时显示已完成
  - 已完成卡片视觉变灰

  主管看板逻辑
  主管页用于体现管理价值。

  统计内容包括：

  - 今日总单
  - 待接单
  - 处理中
  - 已完成
  - 已超时
  - 平均接单时间
  - 平均完成时间
  - 员工完成排行

  可进一步扩展：

  - 部门统计
  - 超时率
  - 高频需求
  - 超时/即将超时列表

  AI 来电模拟逻辑
  目标是展示“电话内容自动生成工单”。

  输入一段中文电话文本，使用本地规则解析：

  输出：

  {
    room_number / room
    summary / request
    department
    priority
    sla_minutes / due_at
    confidence
    extracted_items
  }

  当前第一版可简单解析：

  - 房号
  - 需求
  - 部门
  - 优先级

  部门关键词示例：

  客房部：

  水、矿泉水、毛巾、浴巾、牙刷、拖鞋、被子、枕头、纸巾、洗漱、打扫、卫生、清洁

  工程部：

  空调、电视、热水、灯、门锁、马桶、下水道、漏水、窗户、插座、维修、坏了、不制
  冷、没电

  前台：

  发票、续住、退房、押金、房卡、叫醒、寄存、换房

  餐饮部：

  早餐、送餐、餐厅、外卖、筷子、餐具、咖啡、茶

  主管：

  投诉、不满意、太吵、退款、经理、差评、生气、严重

  优先级规则：

  投诉、不满意、退款、经理、严重、漏水、没电、门锁打不开 => urgent
  尽快、马上、很急、太热、太冷、坏了 => high
  否则 normal

  AI 来电页面要突出：

  当前 Demo 使用文本模拟电话转写结果。
  真实落地时，可由讯飞、阿里云、腾讯云等语音系统把转写文本推送到这里。

  数据层
  第一版可支持两种模式：

  1. Supabase 模式
      - 使用 Supabase PostgreSQL 存工单
      - 使用 Supabase Realtime 监听工单变化
      - 员工端操作后，前台自动更新
  2. 本地 Demo 模式
      - 没有 Supabase 环境变量时使用浏览器本地数据
      - 不白屏
      - 可用 localStorage + BroadcastChannel 模拟多页面同步

  工单核心字段
  当前第一版实际使用字段：

  type Ticket = {
    id: string
    room: string
    request: string
    department: Department
    source: "ai_call" | "frontdesk" | "staff" | "manager"
    status: TicketStatus
    priority: Priority
    assignee_id: string | null
    assignee_name: string | null
    created_at: string
    accepted_at: string | null
    completed_at: string | null
    due_at: string
    notes: string | null
  }

  如果后续做正式系统，建议数据库字段改为更清晰的：

  room_number
  summary
  detail
  sla_minutes
  remark

  但当前回退后的第一版仍使用：

  room
  request
  due_at
  notes

  演示数据
  当前需要包含不同场景：

  - 未接单
  - 已接单/处理中
  - 已完成
  - 超时
  - 客房部长期不超时单

  新增过两个客房部长期不超时演示单：

  1105：需要补充两瓶矿泉水和一套洗漱用品
  1310：客人需要加送一个枕头和一条浴巾

  本地模式如果看不到新增数据，需要在前台点击“重置演示数据”，因为浏览器会优先读旧
  localStorage。

  真实工程建议
  前台看板应避免展示过多技术状态，而是围绕前台真实问题设计：

  有没有人接？
  谁在处理？
  有没有超时？
  有没有完成？
  要不要催？

  所以正式产品中，推荐把前台主视图设计为：

  顶部 KPI 筛选卡片
  统一工单列表
  紧要优先排序
  颜色表达风险和状态

  不建议长期依赖 timeout 作为唯一状态。更好的方式是：

  status 表示流程状态
  overdue/timeout 由 SLA 动态计算

  这样可以避免：

  status = pending 但实际已经超时
  status = timeout 但后来已完成

  这类数据不一致问题。
